UTAGE API利用制限と
テスト運用スケジュール API枠を守りながら、定時取得・ローカルSQL・人レビューで進める

2026/06/26 MTG資料 ｜ 対象：LINE自動回答システム ｜ 前提：UTAGE APIはアカウント共有枠
今日伝えたい結論： UTAGE APIは「APIキーごと」ではなく、1つのUTAGEアカウント全体で利用制限を共有する。 自動回答だけで使い切れる枠ではないため、必要なデータを定時で取得してSQLに蓄積し、回答生成時は基本的にローカルDBを読む方式にする。

1今日の説明ポイント

APIをたくさん叩く設計ではなく、限られた枠を全体で有効に使う設計にする。

API枠はアカウント共有

1つのUTAGEアカウントに対して1日の利用制限がかかる。複数のAPIキーを発行しても、利用枠は分離されない。

回答時はDBを優先

毎回UTAGEから全データを取り直さず、定時取得したデータをSQLに入れて、回答生成ではDBから必要情報を読む。

個人情報はローカル管理

SQLには個人情報が入るため、クラウドには置かず、Mac miniのローカルに置く前提で進める。

補足：1分単位の残量はレスポンスヘッダで確認できる一方、1日単位の残量は現状APIから確認できない。つまり「今日あと何回使えるか」は自前で記録する必要がある。

2UTAGE API利用制限の前提

今回の自動回答は、UTAGE APIを「共有資源」として扱う必要がある。

項目現状わかっていること運用上の意味
制限の単位 1つのUTAGEアカウント全体で共有。APIキーを複数発行しても枠は共通。 自動回答、他ツール、他担当者のAPI利用が同じ枠を使う。
1分あたり レスポンスヘッダで上限・残量を確認できる。 短時間に大量実行しない限り、今回の主なボトルネックにはなりにくい。
1日あたり 日次上限があるが、日次残量はヘッダや取得APIでは見えない。 自分たちでAPI呼び出し回数を記録し、消費量を管理する。
1カウントの基準 API 1コールを1回として扱う。取得件数ではなく呼び出し回数が基準。 取得時はper_page=100など、1回で多く取る設計が効率的。
リスク：他の人や他ツールが同じ日にAPIを多く使うと、自動回答側では残量が見えないまま急に上限に当たる可能性がある。

3今回の取得方針

「使うたびに取りに行く」ではなく、「定時に必要分だけ取り込む」方式にする。

SCHEDULE
定時実行
夜間または低負荷時間帯に、必要な差分だけUTAGE APIから取得する。
→
COLLECT
差分取得
友だち情報、チャット履歴、ラベル、流入情報などを更新分中心に取る。
→
STORE
SQLへ保存
Mac miniのローカルDBに蓄積。クラウドには置かない。
→
ANSWER
回答生成
AIはDBを参照し、過去履歴・ラベル・似た質問を踏まえて回答案を作る。
→
REVIEW
人が確認
LINEチームが確認・修正・送信。採用率と修正理由を蓄積する。

この方式のメリット

  • API消費を最小化できる
  • 回答生成時の速度が安定する
  • 過去回答を横断検索しやすくなる
  • 採用率・不採用理由をあとから分析できる

守るルール

  • APIキーは処理マシンに置く。人に配らない
  • SQL本体はMac miniローカル管理
  • 外部共有する出力は必要最小限にする
  • 429や上限到達時は停止せずログで検知する

4LINEチームに下ろすフォーマット

テスト運用では、既存の返信チェックリストに近い形で出す。現場の動きを大きく変えない。

項目AI/システムが出す内容LINEチームが見るポイント
受信メッセージ読者から届いた本文。必要に応じて直近履歴も添える。質問の意図が正しく拾えているか。
カテゴリ日程、Zoom、特典、個別相談、見送り、不安相談など。自動化できるカテゴリか、人対応に回すべきか。
論点1通の中で答えるべきポイントを分解して表示。抜け漏れがないか。
参照した根拠過去回答、定型文、ラベル、直近履歴など。回答案の根拠が妥当か。
AI回答案そのまま送れる文面に近い下書き。修正して送る／人対応にする／送らない、を判断。
採用結果採用、修正採用、不採用、人対応、返信不要。1〜2か月の分析用データとして残す。
修正理由足りない情報、トーン違い、事実確認不足など。AI改善と自動化可否の判断材料にする。
目標：LINEチームには「AIが考えた理由が見える回答案」として渡す。単なる返信文だけを渡さず、なぜその回答になったかを確認できる形にする。

6MTGで話す文面

そのまま読み上げられるように、話し言葉寄りに整えたもの。

UTAGEのAPI利用制限について共有します。UTAGEのAPIは、APIキーごとではなく、1つのUTAGEアカウント全体に対して1日の利用制限がかかる形です。

つまり、APIキーを複数発行していても、自動回答だけで専用の枠を持てるわけではありません。同じUTAGEアカウントでAPIを使っている人やツールがあれば、すべて同じ枠を共有することになります。

また、1分あたりの残量はヘッダで確認できますが、1日の残量は現状APIから確認できません。そのため、今回の自動回答ではAPIを都度たくさん叩くのではなく、定時で必要なデータを取得し、SQLに保存する方式で進めたいと考えています。

回答生成時は、UTAGE APIへ毎回取りに行くのではなく、基本的にはSQLに蓄積したデータを参照します。これにより、APIの利用回数を抑えながら、過去のやりとりやラベル、過去回答を踏まえた回答案を作れるようにします。

SQLには個人情報が入るため、クラウドには置かず、Mac miniのローカルに置く前提です。外部に共有するのは、LINEチームが確認するために必要な出力だけに絞ります。

スケジュールとしては、6月25日までにAPI仕様、回数のカウント基準、LINEチームに下ろすフォーマットを整理し、7月5日までにテスト実装。7月6日以降、1〜2か月ほどテスト運用しながら採用率や修正理由を蓄積して、自動化できる範囲を判断していく流れです。
